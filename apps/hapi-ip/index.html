Hapi 优选 IP 配置 ​

大多数情况下，HAPI 保持默认网络就能正常使用。只有在远程入口已经跑通，但不同地区访问差异明显、手机接管不稳定、或者你准备长期在线使用时，再继续看这一页。

如果你还没完成基础接入，先看 Hapi 远程控制。
如果你已经能正常远程接入，只是想继续优化访问体验，再看 Hapi 进阶：优选 IP 配置。

先判断是不是 IP 问题 ​

很多人看到“远程体验差”，会直接把问题归到 IP 或线路上。实际上，HAPI 常见问题通常先分成两层：

  1. HAPI 远程入口层 决定手机或浏览器能不能顺利打开会话页面
  2. 底层模型调用层 决定 Claude Code、Codex、OpenCode 自己连到 小蓝中转站 后快不快

可以这样判断：

  • 页面都很难打开：先看 HAPI Hub、Relay、当前网络
  • 页面打开正常，但消息发不出去：先看会话状态和 Runner
  • 页面与消息都正常，只是回复慢：优先回到底层 CLI、模型和上下文大小

也就是说，“优选 IP”主要是在优化第一层，不是替代模型配置。

如果你只想先判断自己该走哪条路线，可以先按使用场景分：

  • 同一 Wi-Fi 下临时接管：优先局域网直连
  • 自己几台设备长期互通：优先固定设备网络互联
  • 给外部浏览器长期访问：优先固定公网入口或稳定隧道

场景先选对，后面才谈得上“优选”。

什么情况下值得继续优化 ​

出现下面这些情况时，再继续折腾入口线路更有意义：

  • 同一条会话，本机访问正常，手机外网访问不稳定
  • 不同地区同事接入同一个 HAPI Hub，体验差异明显
  • HAPI 准备长期跑在远端开发机、云主机或家里常开电脑上
  • 你已经把 Claude Code、Codex 或 OpenCode 单独跑通，只剩远程体验还不够稳

如果只是偶尔临时接管一次，默认链路通常已经够用。

更稳的做法 ​

先固定一台稳定机器 ​

比起在笔记本上临时开一条会话，固定一台更少休眠、少切网的机器会稳很多。

适合长期使用的机器通常有这些特点：

  • 不会频繁合盖睡眠
  • 网络出口比较固定
  • 项目目录和 CLI 配置长期保留

这样做的好处是，HAPI 入口、底层 CLI、工作目录都会更稳定，不需要每次出门前重新搭一遍。

先让最短链路跑通 ​

优选 IP 不是一开始就加很多中间层。更稳的顺序是：

  1. 本机浏览器访问 HAPI
  2. 同一网络下另一台设备访问 HAPI
  3. 手机流量或外网访问 HAPI
  4. 确认长期需求后，再考虑固定入口、反代或其他网络优化

链路越短，定位问题越快。先把最短路径跑通，再去做更复杂的入口设计。

同局域网访问时，最容易忽略的一步是监听地址。
如果 Hub 还只监听 127.0.0.1，那本机能打开并不代表同网设备也能打开。

先证明底层 CLI 已经稳定 ​

继续优化前，最好先单独验证一次底层工具：

bash
claude --version
codex --version
gemini --version
opencode --version

以及至少跑通一条真实会话。原因很简单：如果底层 CLI 自己都还没稳定走 小蓝中转站，继续换入口线路也不会让会话变正常。

一个实用的排查顺序 ​

怀疑远程体验差时，按这个顺序看最省时间：

  1. 先在本机单独运行底层 CLI，确认模型回复正常
  2. 再确认 hapi hub 或 hapi hub --relay 能稳定给出访问入口
  3. 用本机浏览器接一次
  4. 用同局域网设备再接一次
  5. 最后再换外网、热点或长期在线机器

只有前面几步都已经稳定，后面的线路优化才是“提速”；否则很容易把基础配置错误误判成 IP 问题。
