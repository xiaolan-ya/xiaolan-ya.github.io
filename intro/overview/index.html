为什么需要 小蓝中转站 网关？ ​

已经熟悉 OpenAI 兼容接口的用户，可以直接查看 创建专属 Key 和 工具接入指南。

小蓝中转站 解决什么 ​

配置 Claude Code、Gemini CLI、Codex、Cherry Studio 等工具时，常见问题集中在三个方面：多个平台账号、不同接口格式、不同 Key 和账单分散管理。

小蓝中转站 提供统一模型网关，把这些工具接到同一个接口入口。

接入后的请求链路：

txt
你的工具  ←→  小蓝中转站 Endpoint  ←→  可用模型
1

第一次接入时，更容易混在一起的其实还有三件事：

  • 下载工具：去各自官方入口下载安装
  • 创建 Key：在 小蓝中转站控制台生成专属凭证
  • 把工具接到网关：按各工具自己的协议填写地址、Key 和模型

把这三件事分开以后，排障会清楚很多。
很多“工具装好了却不能用”的问题，最后都不是下载层，而是 Key、模型或地址层。

对 Codex、OpenAI SDK 这类直接填写 OpenAI Base URL 的工具，常用配置只需要记住三项：

txt
Base URL: https://xiaolan1.cn/v1
API Key: 控制台创建的 Key
Model: 模型广场复制的模型名
1
2
3

Cherry Studio 也是 OpenAI 兼容思路，但它的服务商地址栏按官方当前说明更适合填写根地址 https://xiaolan1.cn，再由客户端自己继续补后面的兼容路径。

为什么同一个网关，不同工具的地址却不一样 ​

第一次接入最容易踩的坑，就是把所有工具都当成完全一样的 OpenAI 兼容客户端来填。

实际上，不同工具对 Base URL 的要求并不完全一样：

工具地址应该怎么填原因
Codex / OpenAI SDK / 大多数 OpenAI 兼容工具https://xiaolan1.cn/v1工具本身按 OpenAI 兼容接口发请求
Claude Codehttps://xiaolan1.cnClaude Code 自己继续拼 Anthropic 协议路径
Gemini CLIhttps://xiaolan1.cnGemini CLI 自己继续拼 Gemini API 路径
Cherry Studiohttps://xiaolan1.cn客户端自己继续补 OpenAI 兼容路径

只要这一层填错，就很容易出现这类现象：

  • Key 看起来没问题，却一直 404
  • 能启动客户端，却拉不到模型
  • 明明装好了工具，却始终发不出第一条正常请求

为什么不是每个工具都直接连官方 ​

很多用户第一次接触时，会把“下载工具”和“把工具接到哪个网关”混在一起。其实这是两件不同的事：

  • 下载工具：仍然优先从各自官方入口下载
  • 接入模型：通过 小蓝中转站 统一管理地址、Key、模型和账单

这样做的直接好处是：

  1. 不同工具不需要分别记不同平台的账号和充值方式
  2. 更容易按设备、工具、项目拆分 Key
  3. 排障时可以直接在同一个控制台看日志和消耗
  4. 模型名、分组和价格都能在同一个模型广场确认

也就是说，小蓝中转站 不是替代工具本身，而是把“工具怎么接模型”这件事统一起来。

核心体验优势 ​

统一 Endpoint ​

大多数支持 OpenAI 兼容接口的开发工具，只需要配置这一个地址：

txt
https://xiaolan1.cn/v1
1

密钥沙盒化管理 ​

可以按设备、工具或项目分配独立 Key。发现配置泄露或不再需要时，在控制台停用或删除对应 Key。

模型矩阵自由切换 ​

在控制台的“模型广场”查看当前可用模型、模型名和费率，复制完整模型名后写入客户端：

https://xiaolan1.cn/pricing

账单所见即所得 ​

余额和消费明细按人民币计算。控制台日志可以按 Key、模型和时间范围查看请求消耗。

统一入口不等于所有工具都用同一套参数 ​

第一次接入最容易踩的坑，就是把所有工具都当成完全一样的 OpenAI 兼容客户端来填。

大多数工具确实使用：

txt
https://xiaolan1.cn/v1
1

但 Claude Code 是例外。Claude Code 读取的是 Anthropic 协议变量，Base URL 需要填写根地址：

txt
https://xiaolan1.cn
1

不要给 Claude Code 手动补 /v1。
Codex、OpenAI SDK 这类 OpenAI 兼容工具，继续填写 https://xiaolan1.cn/v1。
Cherry Studio 则更适合填写根地址 https://xiaolan1.cn，让客户端自己补路径。

Gemini CLI 当前版本也是例外。它读取的是 GOOGLE_GEMINI_BASE_URL 和 GEMINI_API_KEY，地址填写根地址 https://xiaolan1.cn，再由 CLI 自己拼接 /v1beta/models/... 这类 Gemini API 路径。

把这一点分清楚，能少掉很多“明明 Key 没问题，却始终连不上”的排障时间。

跑通第一条链路以后，排障为什么会轻很多 ​

当你已经让某一个工具先正常回复过一次，后面再遇到问题时，范围通常就能缩小到：

  • 当前换的模型是否超出这把 Key 的分组或模型权限
  • 当前项目目录是否有额外配置覆盖默认值
  • 当前终端是否读到了新的环境变量
  • 当前工具是否还停留在旧服务商、旧模型或旧会话

这也是为什么文档一直建议先选一条最短路径：

  • 先聊天试通接口：Cherry Studio
  • 先在终端里改代码：Claude Code
  • 已经熟悉 OpenAI CLI：Codex
  • 想要长上下文终端助手：Gemini CLI

Key 安全 ​

  • API Key 只保存在本地环境变量、.env 文件、可信客户端或密码管理器中。
  • 不要把包含 Key 的配置文件提交到 Git 仓库。
  • 共享服务器、测试脚本和公网服务建议使用单独 Key，并设置额度上限。

跑通后你会得到什么 ​

当某个工具已经能通过 小蓝中转站 正常回复时，通常说明下面几件事已经同时成立：

  • 当前网络能够访问 小蓝中转站 网关
  • 你的账号和余额状态正常
  • 这把 Key 处于可用状态
  • 当前分组和模型权限允许调用目标模型
  • Base URL、API Key、Model 至少这三项没有填错

这也是为什么文档一直强调先跑通一条最小链路。只有先证明基础链路是通的，后面继续换模型、换工具、换客户端才会更稳。

常见接入方式 ​

  • 图形界面聊天：Cherry Studio
  • 终端编程：Claude Code
  • OpenAI CLI 工作流：Codex
  • 长上下文终端助手：Gemini CLI
  • 自己写脚本：OpenAI 兼容接口
