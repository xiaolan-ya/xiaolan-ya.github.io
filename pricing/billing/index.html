计费说明 ​

  • 余额与消费按元显示
  • 模型与单价可在模型广场查看
  • 充值和消费记录在控制台查看

模型广场：https://xiaolan1.cn/pricing

计费最容易让新手困惑的地方，不是“要不要收费”，而是“为什么同样一句话，不同模型、不同工具、不同上下文长度，费用会不一样”。核心原因通常有三类：

  • 你选的模型不同
  • 输入和输出的 token 数不同
  • 你一次带进去的上下文大小不同

可以先记一个最实用的判断：

txt
模型越强、上下文越大、输出越长，费用通常越高
1

看价格先看模型广场 ​

不同模型的价格、能力和可用状态会变化。实际使用前，以模型广场展示为准。

复制模型名

进入 模型广场，用搜索框找到目标模型，点击模型名复制，粘贴到工具里的 model 字段。模型名以模型广场显示为准，不要手打。

建议关注：

  • 模型名称
  • 输入价格
  • 输出价格
  • 是否支持图像、音频、长上下文
  • 当前是否可用

第一次接入时，不建议直接按照“最强模型”来选。先拿轻量模型把链路跑通，再根据任务换到更强模型，通常更省钱也更容易排障。

第一次不确定选什么时，可以按这个顺序走：

  1. 先选一个轻量模型
  2. 发一句“你好”或一个最短测试任务
  3. 看日志里的 token 和扣费
  4. 再决定是否需要切到更强模型

消耗在哪里看 ​

控制台里的日志和统计页可以看到请求明细。

常见字段：

  • 请求时间
  • 使用的 Key
  • 调用模型
  • 输入 Token
  • 输出 Token
  • 实际扣费
  • 请求状态

如果你按工具拆分 Key，就能很快看出哪个工具消耗最高。

真正开始长期使用后，最值得看的通常不是“总共花了多少”，而是：

  • 哪把 Key 消耗最高
  • 哪个模型消耗最高
  • 哪一类任务最容易把上下文带大

为什么同一个任务有时费用差很多 ​

下面这些情况都会明显影响成本：

  • 把整个仓库、整篇长文一次性丢给模型
  • 多轮对话带了很长历史
  • 同时请求多个结果或更大尺寸的图片
  • 自动化脚本出现重试循环

所以“模型单价”只是起点，真正的消耗还和你怎么喂上下文、怎么拆任务密切相关。

举个最常见的差别：

  • 只发一句“修这段报错”
  • 把整个仓库、十几段日志和几轮长历史一起发进去

哪怕用的是同一个模型，后者的 token 也可能明显更高。

降低成本的做法 ​

  • 日常问答用轻量模型
  • 复杂代码任务再切强模型
  • 不要把整个仓库一次性塞给模型
  • 长文先摘要，再让强模型处理关键段落
  • 自动化脚本必须设置额度上限

如果你已经开始同时用多个工具，通常更稳的做法是：

  • 桌面聊天一把 Key
  • CLI 编程一把 Key
  • 自动化脚本一把 Key

这样做不只是为了看日志更清楚，也方便你对高消耗场景单独限额。

先看账户余额，再看 Key 限额 ​

很多人以为“还有余额”就代表所有请求都应该成功。实际上还要再看一层：

  • 账户余额：整个平台账户还剩多少钱
  • Key 限额：这把 Key 被单独限制到什么程度

因此：

  • 余额充足，但某把 Key 到达上限时，请求仍会失败
  • Key 额度还没满，但账户总余额不足时，请求也会失败

排障时，最好把钱包和令牌设置一起看。

也可以简单理解成：

txt
钱包余额 = 总预算
Key 限额 = 某一把 Key 的局部预算
1
2

余额异常排查 ​

  1. 先看日志，按时间倒序查看最近请求。
  2. 按 Key 过滤，找出消耗来源。
  3. 发现异常请求，先禁用对应 Key。
  4. 检查脚本、客户端或自动化任务是否循环调用。
  5. 必要时联系售后，并提供截图和时间范围。

截图前记得打码完整 Key。
