协议兼容声明 ​

支持 OpenAI Compatible API（OpenAI 兼容接口）的工具，通常都可以通过自定义 Base URL 和 API Key 接入 小蓝中转站。

这一页主要面向：

  • Cherry Studio
  • Codex
  • 自定义脚本
  • OpenAI SDK
  • 支持 OPENAI_BASE_URL 或同类字段的客户端

这一页不适用于：

  • Claude Code
  • Gemini CLI 当前版本

如果你用的是 Claude Code，查看 Claude Code 部署。
如果你用的是 Gemini CLI，查看 Gemini CLI 部署。

唯一 Endpoint ​

OpenAI 兼容入口统一使用：

txt
https://xiaolan1.cn/v1
1

常见字段名可能写成：

txt
Base URL
API Base
Endpoint
OpenAI Base URL
OPENAI_BASE_URL
1
2
3
4
5

看到这些字段，都填 https://xiaolan1.cn/v1。

这里填的是 接口根地址，不是网页地址，也不是某个具体接口的完整路径。
最稳的写法，就是停在 /v1 这一层。

不要填：

txt
https://xiaolan1.cn
https://xiaolan1.cn/dashboard/overview
https://xiaolan1.cn/pricing
https://xiaolan1.cn/v1/chat/completions
1
2
3
4

前面三个是网页地址，最后一个是具体接口路径，都不适合作为 Base URL。

地址填错后的常见现象：

  • 保存配置时看起来没问题，但一发请求就报错
  • 返回网页内容，而不是 JSON
  • 客户端提示 404、Not Found 或 endpoint 不存在
  • 模型列表始终拉不下来

遇到这类现象时，先把地址改回 https://xiaolan1.cn/v1，再继续查 Key 和模型名。

支持的核心能力范围 ​

小蓝中转站 会把多家模型能力映射到常见 OpenAI 兼容接口：

  • 对话补全 (Chat Completions)：最高频的场景，支持流式输出。请参阅 /v1/chat/completions 规范。
  • 文本补全 (Completions)：适用于代码补全工具。请参阅 /v1/completions 规范。
  • 向量生成 (Embeddings)：适用于本地 RAG 和知识库检索。请参阅 /v1/embeddings 规范。
  • 其他周边能力：请查阅左侧 API 详细参考手册。

只要工具本身支持改 baseURL、base_url、OPENAI_BASE_URL 或同类字段，一般都能按这套方式接入。

鉴权写法 ​

标准 Header：

txt
Authorization: Bearer sk-你的专属Key
1

很多桌面客户端会自动补 Bearer，输入框里通常只需要粘贴 Key。
自己写脚本时，Header 要完整。

环境变量接法 ​

很多 CLI 工具和脚本，除了图形界面填地址，也支持直接从环境变量读取：

bash
export OPENAI_BASE_URL="https://xiaolan1.cn/v1"
export OPENAI_API_KEY="sk-你的专属Key"
1
2

这样做的好处是地址和 Key 分开管理：

  • 改 Key 时不用回头重写脚本
  • 多个脚本可以共用同一套环境
  • 出问题时更容易判断是地址错了，还是 Key 错了

Windows PowerShell：

powershell
[Environment]::SetEnvironmentVariable("OPENAI_BASE_URL", "https://xiaolan1.cn/v1", "User")
[Environment]::SetEnvironmentVariable("OPENAI_API_KEY", "sk-你的专属Key", "User")
1
2

最小验证 ​

第一次联调时，优先用最小请求证明四件事：

  1. 地址写对了
  2. Key 能鉴权
  3. 当前网络能打到网关
  4. 模型名是当前 Key 真能使用的那个

先列模型：

bash
curl https://xiaolan1.cn/v1/models \
  -H "Authorization: Bearer sk-你的专属Key"
1
2

如果能返回 JSON，说明地址、Key 和网络通常已经没有大问题。

再做一次最小对话测试：

bash
curl https://xiaolan1.cn/v1/chat/completions \
  -H "Authorization: Bearer sk-你的专属Key" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "从模型广场复制的模型名",
    "messages": [
      { "role": "user", "content": "你好" }
    ]
  }'

模型名来源 ​

模型名以 模型广场 为准。

不要直接沿用官方平台示例里的模型名，也不要手敲一个自己觉得“差不多”的名字。
小蓝中转站 会统一展示当前可用模型和价格，复制页面里的完整模型名更稳。

第一次接入时，建议先选一个轻量模型验证链路。
能正常返回后，再切到长上下文、代码能力更强或更贵的模型。

兼容但不等于完全相同 ​

OpenAI 兼容接口解决的是请求格式问题。不同上游模型本身仍然可能有差异：

  • 上下文长度不同
  • 是否支持图像输入不同
  • 是否支持工具调用不同
  • 是否支持流式输出不同
  • 计费方式不同

生产脚本上线前，先用目标模型跑一遍最小测试，再决定默认模型和超时策略。
