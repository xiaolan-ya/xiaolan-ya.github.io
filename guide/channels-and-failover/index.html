渠道与节点切换 ​

小蓝中转站 在网关侧处理渠道切换和故障转移。客户端侧最稳的做法，是把已经配置好的地址保持不变。

在日常使用中，不同上游模型厂商偶尔会出现区域性波动、单条渠道失效、某个时段超时增多。
这种时候，真正需要变动的通常不是你本地写好的 Endpoint，而是网关内部到底该选哪条可用渠道。

常见工具对应的地址保持如下：

工具类型保持使用的地址
OpenAI 兼容工具https://xiaolan1.cn/v1
Claude Codehttps://xiaolan1.cn
Gemini CLI 当前版本https://xiaolan1.cn

很多用户遇到响应变慢、单次超时，第一反应是去客户端里换地址、改 endpoint，甚至把别的教程里的 URL 贴进来。对于已经接到 小蓝中转站 的工具来说，这通常不是优先处理方式。

智能路由与高可用 ​

一次请求进入网关后，大致会经历这几层判断：

  1. 先按分组和模型筛选：当前 Key 所在分组、模型权限、渠道可用状态，先决定“哪些渠道有资格被选中”。
  2. 再按优先级选渠道：同一分组下，同一模型可能挂了多条渠道。网关会优先尝试当前优先级更高的可用渠道。
  3. 遇到可重试错误再换下一条：如果这次请求返回的是可重试错误，网关才会继续尝试下一条可用渠道。
  4. auto 分组时可继续跨分组：如果你的 Key 走的是 auto 分组，并且当前配置允许跨分组重试，当前分组的可用优先级用完后，才会继续尝试下一个分组。

这意味着：同一把 Key、同一个已配置地址，网关会尽量处理“同模型、同权限范围内”的底层通道波动。
客户端侧更值得优先检查的，通常是模型、Key、上下文大小和超时设置。

开发者需要做什么？ ​

OpenAI 兼容脚本或客户端，通常保持下面这类地址配置即可：

bash
export BASE_URL="https://xiaolan1.cn/v1"
1

如果你使用的是 Claude Code 或 Gemini CLI，就继续保留它们各自文档里的根地址配置，不要因为某次变慢就临时改成别的路径。

真正需要你手动处理的，通常只有三类：

  • 换模型：当前模型维护、拥堵，或这个模型本身就只有少量上游时，切同类模型通常更快。
  • 换 Key：某把 Key 被禁用、泄露或额度耗尽。
  • 缩上下文：请求太大导致慢或失败。

故障转移能解决什么，不能解决什么 ​

故障转移主要解决的是“底层某条渠道暂时不稳”这类问题，但不是所有错误都能靠切通道修好。

更可能由故障转移缓解的问题 ​

  • 单条渠道临时不可用
  • 503 / 504 这类上游不可用或超时
  • 单次偶发超时
  • 同一模型在某个时段明显变慢，但换同模型的其他渠道后可恢复

通常不是节点问题 ​

  • 401：Key 错误、余额不足或 Key 被禁用
  • 400：请求格式或模型名错误
  • 404：模型名不存在
  • 429：频率或额度限制
  • 账户或分组本身没有这个模型权限

这些错误继续改 endpoint，通常不会变好。

遇到大规模不可用 ​

如果在模型广场或状态接口看到大面积报错，可以先：

  1. 在客户端内暂时切至其他厂商的同级别模型（如从 Claude 切换到 Gemini）。
  2. 前往 状态探针页面 查看整体服务是否正常。

如果只是单个模型不稳，不必第一时间重做整套配置。先换同类轻量模型验证，通常更快。

状态探针更适合看“站点是不是整体可用”，不适合拿来证明“某个具体模型这一刻一定正常”。
单个模型的拥堵、维护或上游限流，还是要回到模型本身来判断。

客户端超时怎么设 ​

如果你的工具允许设置超时时间，不要设得太短。复杂模型和长上下文任务可能需要更久。

建议：

  • 普通聊天：60 秒以上
  • 代码库分析：120 秒以上
  • 图像、音频、视频：按任务类型放宽

超时设得过短时，网关还没来得及完成同模型的重试，客户端就已经先断开了。
表面看像“整条链路全挂了”，实际可能只是你给的等待时间太短。

一个更省时间的判断顺序 ​

遇到问题时，优先按这个顺序判断：

  1. 先看是不是 401 / 404 / 429 这类明确错误
  2. 再看状态探针和模型广场是否有整体异常
  3. 再换同类轻量模型做最小验证
  4. 最后才考虑是不是当前模型所在渠道不稳

这样能避免把“Key 没权限”或“模型名写错”误判成节点问题。
